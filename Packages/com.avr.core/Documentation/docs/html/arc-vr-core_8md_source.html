3 # arc-vr-core {#arc-vr-core}
4 
5 Arc-vr-core is the core package of the arc-vr framework. It contains all base classes and components that the other packages make use of. These include all rudimentary VR features such as spatial tracking, controller-inputs and other utilities. In many cases, this package might be all you need for your project.
6 
7 Classes are intentionally built to be inherited, derived and modified. Don't shy away from taking a look into the source code and adapting it to better suit your project.

11 ## Documentation {#Documentation}
12 
13 The documentation you are looking at right now, as well as docs for all other packages are included in arc-vr-core.
14 If you take a look inside `Packages/com.avr.core/Documentation/docs` you will find all html, css and resource files of the docs.
15 
16 Each arc-vr component will have a button inside the Unity Editor that opens the respective documentation page in the browser (See (Editor Untilities)[@ref EditorUtilities]). If you are creating your own `AVR_Component`-objects and want them to link to the proper page, take a look at the (DocumentationUrl attribute)[@ref DocumentationUrlAttribute] to see how to set this link yourself.
17 
18 These files are generated using [Doxygen 1.8.7](https://www.doxygen.nl/index.html). Under `Packages/com.avr.core/Documentation` you will find the doxygen executable as well as the respective [doxygen configuration file](https://www.doxygen.nl/manual/config.html) called `avr_dxy_config`.
19 
20 Tutorials (such as this one) are located in `Documentation/tutorials`. In these markdown files you can use all default doxygen commands as well as a few custom commands that are defined in the config file.
21 
22 You can re-generate the whole projects documentation any time by simply running `.\doxygen_1_8_7.exe .\avr_dxy_config` inside the Documentation folder. Doxygen will automatically extract code-summaries and produce up-to-date html files.

24 ## Attributes
25 
26 arc-vr-core includes several [C# Attributes](https://docs.microsoft.com/en-us/dotnet/csharp/programming-guide/concepts/attributes/) for various purposes. For a more detailed description head to the @ref AVR.Core.Attributes documentation.
27 
28 Here's a quick overview:
29 
30 ## ConditionalHideInInspector
31 
32 This attribute lets you hide properties in the Unity inspector based on a condition. Here's a basic example:
33 
34  public bool tracking = false;
35 
36  [ConditionalHideInInspector("tracking", true)]
37  public string type;
38 
39 The property 'type' will not be shown in the inspector unless 'tracking' is set to true.
40 
41 ![img](../../res/images/core_att_condhide.jpg)
42 
43 You can also use this attribute with more sophisticated conditions than just booleans. Check the @ref AVR.Core.Attributes.ConditionalHideInInspector "detailed docs" for more.
44 
45 ## ReadOnly
46 
47 Makes a propery read-only in the Unity inspector. Example: (target can not be set in the inspector, but still viewed)
48 
49  [ReadOnly]
50  public string target="sample_target";
51 
52 ![img](../../res/images/core_att_readonly.jpg)
53 
54 ## FoldoutGroup
55 
56 Creates a foldout group of properties in the Unity inspector using a string-based group-id.
57 
58 \note Even if a group-id is the same, a new group will be created unless the group-members are defined subsequently (see example below).
59 
60 Example:
61 
62  [FoldoutGroup("group1")]
63  public float num1 = 30f;
64  [FoldoutGroup("group1")]
65  public float num2 = 0.5f;
66 
67  [FoldoutGroup("group2")]
68  public string abc = "abcdefg";
69 
70  [FoldoutGroup("group1")]
71  public float num3 = 12f;
72 
73 ![img](../../res/images/core_att_foldout.jpg)
74 
75 ### ConsoleCommand
76 
77 With this attribute you can declare a static, void function to act as a command for the @ref AVR.Core.AVR_DevConsole. The parameters of this function have to be either of type 'string', 'string[]' or none.
78 
79  [ConsoleCommand("example_command", 2, "Replies with the second argument passed")]
80  static void example_command(string[] args) {
81  print(args[1]);
82  }
83 
84 The example above will create a command called 'example_command' with a minimum of 2 arguments that simply outputs the second argument passed.
85 
86 ### DocumentationUrl
87 
88 This attribute defines the html page that is opened when the user clicks on the documentation button of an AVR_Behaviour in the Unity Editor. The html files are located in the `Packages/com.avr.core/Documentation/html` directory.
89 
90 Example:
91 
92  [DocumentationUrl("class_a_v_r_1_1_core_1_1_a_v_r___controller.html")]
93  public class AVR_Example : AVR_Behaviour
94  { ... }

96 ## Base classes
97 
98 ### AVR_Behaviour
99 
100 Most arc-vr scripts inherit from @ref AVR.Core.AVR_Behaviour "AVR_Behaviour". An AVR_Behaviour is, in essence, a regular MonoBehaviour but with a DocumentationUrl button and some quick-access members such as `AVR_Behaviour.playerRig` and `AVR_Behaviour.root`.
101 
102 ### AVR_Component
103 
104 Components such as feature-providers, tracked devices or modules are represented by an AVR_Component. An AVR_Component is an adds further complexity to AVR_Behaviour. If the @red arc-vr-net package is present, MLAPI-related information can be accessed through `AVR_Component.networkAPI`.
105 
106 \note Each \ref AVR.Core.AVR_Component "AVR_Component" script displays 2-3 buttons within the editor:
107 ![img](../../res/images/core_editorButtons.jpg)
108 - *Documentation Link*: Will open up the respective documentation page in your browser.
109 - *Custom Monobehaviour Events*: Will let you set UnityEvents that are executed when the Monobehaviour Awake, Start, OnEnable or OnDisable are called. You can use this, for instance, to maintain two modules mutually exclusive. For example, you might not want to let the player teleport/move whilst interacting with UI. Then you would simple add an OnEnable event onto your UIInteractionProvider that disables the MovementProvider, and vice-versa for OnDisable.
110 - *Network Behaviour*: This button is only present if the [arc-vr-net](@ref arc-vr-net) package is included in the project. It gives you some useful tools for how components ought to behave when on a network.
111 
112 ### AVR_ControllerComponent
113 
114 An AVR_Component that includes convenient, quick access to an AVR_Controller that is attatched to a parent gameobject.
115 
116 For instance, you can easily execute an action if the trigger-button on a controller is pressed like so:
117 
118  if(controller.inputManager.triggerButton) {
119  // do something...
120  }
121 
122 ### AVR_GenericXRDevice
123 
124 Represents a tracked object, such as a HMD or controller. The track object is identified through the UnityEngine.XR.XRNode enum.
125 
126 Tracking can be enabled/disabled at will and smoothed to avoid jitter.
127 
128 ### AVR_Controller
129 
130 Typical handheld VR controllers, such as the HTC Vive or Vive Index controllers are represented by the AVR_Controller class.
131 
132 The class gives access to input data (if an inputManager is attatched) as well as haptic capabilites.
133 
134 ### AVR_PlayerRig
135 
136 The overall configuration of HMD and controllers (ie the "player") is represented by an AVR_PlayerRig.
137 
138 The playerRig acts as a singleton, meaning, only one can be active in a scene at any given time.
139 
140 The class provides a plethora of useful data and functions, all easily accessible through the AVR_Behaviour.playerRig member. For a full list, see @ref AVR.Core.AVR_PlayerRig.
